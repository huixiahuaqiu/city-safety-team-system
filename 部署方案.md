# 城市安全团队系统 · 生产落地部署方案（自建服务器 + 大文件 + 安全）

> 目标：把现有系统稳定部署到一台（可扩到多台）自建 Linux 服务器，支持**后续持续上传大量大文件**，
> 并覆盖安全、备份、监控、发布回滚、容量规划全链路。所有配置均可直接照抄改值落地。

---

## 0. 先说结论（TL;DR）

- **架构不推翻**：保持「Nginx（对外）→ Python 网关 working_proxy（/api）→ Supabase KV（结构化数据）+ MinIO/磁盘（大文件）」。
  你现在的 `working_proxy.py` 已经实现了分片续传、MD5 秒传、MinIO 落盘、审计日志、目录穿越防护，底子够用，只需**生产化加固**。
- **大文件走两条道**：
  1. 中小文件（≤200MB）：现有 `/api/shared-file/upload`（一次性 multipart）——但要改成流式，别再整包进内存。
  2. 大文件（GB 级）：走现有 `/api/dataset/*` 分片续传，或**浏览器直传 MinIO（预签名 URL）**，绕开 Python 内存。
- **对外只开 80/443**：MinIO、Supabase、Python 网关全部只绑内网/本机，Nginx 统一入口。
- **数据落三处、备三份**：Postgres（Supabase）、对象存储（MinIO）、注册表 JSON —— 各自定时备份 + 异地。
- **磁盘是头号风险**：大文件会把盘吃满。独立数据盘 + LVM + 用量告警必须先做。
- **前端头号工程债**：`index.html` ≈ 3 万行过长——**保持 SPA，按模块外置**（见 §14），禁止拆成多页面。

## 1. 现状盘点（基于代码核对，不是猜）

| 能力 | 现状 | 位置 |
|------|------|------|
| 前端 | 单页 SPA，`index.html` ≈ 2.97 万行 / 1.7MB，8 个外部 CDN | `123123/index.html` |
| 网关 | Python `ThreadingHTTPServer`，静态 + /api 一体 | `123123/working_proxy.py` |
| 结构化数据 | Supabase KV（复用 `patents` 表，`classification=__APP_SYNC__`），前端 15s/5s 轮询同步 | 前端 + 网关 |
| 共享文件 | multipart → 磁盘或 MinIO，注册表 `uploads/shared/_registry.json` | `save_shared_upload` |
| 数据集大文件 | 分片续传 init/chunk/complete，MD5 秒传，默认上限 10GB，分片 2MB | `/api/dataset/*` |
| 标注文件 | 二进制上传 + 打包导出 + 云端分片共享 | `/api/annotation/*` |
| 鉴权 | 静态 token（`X-Upload-Token`/`MLOPS_TOKEN`），`secrets.compare_digest` 常量时间比较 | `check_*_token` |
| 审计 | `logs/server_audit.log`，结构化事件 | `audit_event` |
| 安全基础 | 路径穿越防护、扩展名白名单、大小上限、CORS | 多处 |

**已发现的生产级隐患（本方案会逐一解决）：**
1. `do_POST` 用 `self.rfile.read(content_length)` **整包读进内存**——200MB 上传 = 单请求占 200MB RAM，并发几个就 OOM。大文件必须流式/分片/直传。
2. CORS 为 `Access-Control-Allow-Origin: *`——生产要收敛到自己域名。
3. 静态与上传文件**同源同端口**，下载若不强制 `attachment` 有 XSS/挂马风险（HTML/SVG 直接被浏览器执行）。
4. 静态 token 是「一把钥匙全员共用」，无法按人审计/吊销——中期应换成随登录态签发的短期令牌。
5. `http.server` 是单进程多线程，GIL 下 CPU 密集（打包 zip、OCR）会拖慢全站——建议网关只做 IO，重活异步化。

## 2. 目标架构

```
                       公网/内网用户
                            │  443(HTTPS)
                            ▼
                 ┌─────────────────────┐
                 │        Nginx        │  统一入口
                 │  · 静态直出(前端)     │  · TLS 终止 · 安全头
                 │  · /api 反代 → 网关   │  · 限流 · 大文件缓冲策略
                 │  · /files 下载(强制附件)│
                 └─────────┬───────────┘
             ┌─────────────┼───────────────────────┐
             ▼             ▼                         ▼
   127.0.0.1:8000     127.0.0.1:9000          (外网/自建)
   Python 网关        MinIO(S3 对象存储)       Supabase
   working_proxy      大文件 bucket           Postgres KV
   (只做 IO/校验)      + 预签名直传             (结构化数据)
             │
             ▼
   /data/uploads(本地磁盘兜底) + *_registry.json
```

原则：
- **Nginx 是唯一对外入口**；网关/MinIO/DB 一律内网。
- **前端静态由 Nginx 直出**（比 Python 快一个数量级），Python 只处理 `/api`。
- **大文件优先对象存储**（MinIO），磁盘只作兜底与临时分片。

## 3. 服务器选型与容量规划（大文件是重点）

### 3.1 起步规格（单机可跑，够几十人团队 + TB 级文件）
- CPU：4~8 核（zip 打包/校验吃 CPU）
- 内存：16GB（MinIO 4~6GB + 网关 + 系统缓存）
- 系统盘：60GB SSD（OS + 程序 + 日志）
- **数据盘：独立挂载，起步 1~2TB，用 LVM 便于在线扩容**（大文件全放这里）
- 带宽：按下载峰值估，10~50Mbps 起，大量下载再上 CDN/限速
- 系统：Ubuntu 22.04 LTS 或 Debian 12（长期支持、生态稳）

### 3.2 磁盘布局（关键，先规划再上线）
```bash
# 数据盘用 LVM，方便后续加盘扩容不停机
/data                      # 独立数据盘挂载点(xfs/ext4)
├── minio/                 # MinIO 对象数据
├── uploads/               # 网关本地兜底 + 临时分片
│   ├── shared/
│   ├── datasets/
│   └── annotations/
├── backups/               # 本地备份暂存(再推异地)
└── pg/                    # 若自建 Supabase/Postgres
```
容量公式：`预计文件总量 × (1 + 30%冗余) × (副本数)`；MinIO 单机 EC 关掉时副本=1，重要数据建议多盘或异地再存一份。

### 3.3 内核/文件系统调优
```bash
# /etc/sysctl.d/99-app.conf
fs.file-max = 2097152
net.core.somaxconn = 1024
net.ipv4.tcp_tw_reuse = 1
vm.swappiness = 10
# 生效：sysctl --system
# ulimit：/etc/security/limits.conf  →  * soft/hard nofile 1048576
```

```bash
# start_web.py 默认绑 0.0.0.0:8000，生产应只绑本机。两种做法二选一：
#  (a) 改 run_server 绑 127.0.0.1（推荐）

sudo systemctl status citysafe-gateway
```

> ⚠️ 建议把 `working_proxy.run_server` 里 `server_address=('', port)` 改成 `('127.0.0.1', port)`，
> 让网关只监听本机，彻底杜绝绕过 Nginx 直连。

### 5.4 让上传更省内存（大文件必改）
现状 `self.rfile.read(content_length)` 整包进内存。落地建议按优先级：
1. **大文件一律走 `/api/dataset/*` 分片**（已实现，单请求只 ~8MB，内存安全）。前端「共享文件」超过阈值（如 50MB）时自动切到分片管线。
2. **或浏览器预签名直传 MinIO**（见 6.4），Python 完全不碰文件体。
3. `/api/shared-file/upload` 保留给中小文件，并在 Nginx 侧用 `client_max_body_size` 卡死上限。

### 6.4 GB 级文件的推荐链路：浏览器直传（预签名）
Python 网关只负责「发预签名 URL」，文件体不经过网关，最省资源、最快：
```
前端 → 网关 POST /api/shared-file/presign {name,size,type}
        网关校验权限/大小/扩展名 → 用 SERVICE_ROLE 生成 presigned PUT URL(限时300s)
前端 → PUT 直传 MinIO(经 Nginx /minio-upload 反代)
前端 → 网关 POST /api/shared-file/confirm {objectKey,md5} → 写注册表
```
> 需要新增两个网关接口（presign / confirm）与 Nginx 一个反代 location。这是后续迭代项，
> 现阶段大文件先用已实现的 `/api/dataset/*` 分片续传即可跑通。

## 7. Nginx 配置（静态 + /api 反代 + 大文件 + 安全头）

```nginx
# /etc/nginx/sites-available/citysafe.conf
# ---- 限流区（防爆刷/爆破） ----
limit_req_zone $binary_remote_addr zone=api:10m rate=20r/s;
limit_req_zone $binary_remote_addr zone=login:10m rate=5r/m;

server {
    listen 80;
    server_name your-domain.com;
    return 301 https://$host$request_uri;      # 强制 HTTPS
}

server {
    listen 443 ssl http2;
    server_name your-domain.com;

    ssl_certificate     /etc/letsencrypt/live/your-domain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/your-domain.com/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    ssl_session_cache shared:SSL:10m;

    root /opt/citysafe/current;                # 见第12节：软链到当前版本
    index index.html;

    # ---- 安全响应头（官网级，但适配本项目，先不强上死 CSP） ----
    add_header X-Frame-Options SAMEORIGIN always;
    add_header X-Content-Type-Options nosniff always;
    add_header Referrer-Policy strict-origin-when-cross-origin always;
    add_header X-XSS-Protection "1; mode=block" always;
    # CSP 先用 Report-Only 观察，避免打断内联脚本/CDN/Supabase；整改内联后再切强制
    add_header Content-Security-Policy-Report-Only
      "default-src 'self'; script-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net https://cdn.tailwindcss.com; connect-src 'self' https://*.supabase.co; img-src 'self' data: blob:; style-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net; font-src 'self' data:;" always;

    # ---- 前端静态：HTML 不缓存，带哈希资源长缓存 ----
    location / { try_files $uri $uri/ /index.html; add_header Cache-Control "no-cache"; }
    location ~* \.(js|css|png|jpg|jpeg|gif|webp|svg|woff2?)$ {
        expires 30d; add_header Cache-Control "public, immutable";
    }

    # ---- API 反代到 Python 网关 ----
    location /api/ {
        limit_req zone=api burst=40 nodelay;
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        # 大文件/分片上传：关缓冲、放大超时
        proxy_request_buffering off;
        proxy_buffering off;
        proxy_read_timeout 600s;
        proxy_send_timeout 600s;
        client_max_body_size 220m;             # 覆盖一次性上传上限；分片模式此值可小
    }

    # ---- 登录接口更严限流 ----
    location = /api/login { limit_req zone=login burst=3 nodelay; proxy_pass http://127.0.0.1:8000; }

    # ---- 下载强制附件下载，杜绝 HTML/SVG 被当页面执行 ----
    location /api/shared-file/download {
        proxy_pass http://127.0.0.1:8000;
        add_header Content-Disposition "attachment" always;
        add_header X-Content-Type-Options nosniff always;
    }

    # ---- 敏感文件/目录禁止访问 ----
    location ~ /\.(git|env|ht) { deny all; }
    location ~* \.(py|pyc|sqlite|db|log|bak|example)$ { deny all; }

    access_log /var/log/nginx/citysafe.access.log;
    error_log  /var/log/nginx/citysafe.error.log warn;
}
```
```bash
sudo ln -s /etc/nginx/sites-available/citysafe.conf /etc/nginx/sites-enabled/
sudo nginx -t && sudo systemctl reload nginx
```

## 9. 数据层（Supabase / Postgres）

现在用 Supabase 云。两种落地路线：

- **A. 继续用 Supabase 云（省事，推荐先用）**
  - `anon key` 前端必然暴露，**真正安全边界是 RLS 行级权限**，不是藏 key。
  - 必做：为 `patents`（同步表）开 RLS，按登录身份限制读写；`service_role key` 只放服务端 `.env`。
  - 备份：用 `pg_dump` 定时导出（见第 10 节），或开 Supabase 自带的 PITR。

- **B. 自建 Supabase / Postgres（数据不出内网，合规更强）**
  - Docker 部署 `supabase/postgres` + PostgREST；只绑内网。
  - 运维成本更高（备份、升级、监控自己扛），团队有 DBA 再上。

> 迁移提醒：当前结构化数据借用 `patents` 表存 KV，长期建议**独立出专用表**（application/notice/weekly 等），
> 加索引与 RLS，别把所有东西塞一张表——这是技术债，可排到中期。

## 10. 备份与容灾（3-2-1：3 份、2 种介质、1 份异地）

### 10.1 要备什么
| 数据 | 路径/来源 | 频率 |
|------|-----------|------|
| 对象存储 | MinIO `team-shared` 桶 | 每日增量 + 每周全量 |
| 本地兜底文件 | `/data/uploads/**` | 每日 |
| 注册表/状态 | `*_registry.json`、`mlops_store.json` | 每小时（小而关键） |
| 结构化数据 | Supabase/Postgres `pg_dump` | 每日 |
| 配置 | `.env`、Nginx conf、systemd unit | 变更即备（加密离线存） |
| 审计日志 | `logs/server_audit.log` | 随日志轮转归档 |

### 10.2 备份脚本（cron）
```bash
# /opt/citysafe/backup.sh  (chmod 700, appsvc)
set -euo pipefail
TS=$(date +%F_%H%M)
DST=/data/backups/$TS
mkdir -p "$DST"
# 1) 注册表 + 配置(体积小)
cp /data/uploads/*/_registry.json "$DST"/ 2>/dev/null || true
cp /opt/citysafe/app/mlops_store.json "$DST"/ 2>/dev/null || true
# 2) 对象存储镜像到异地(另一台/另一盘/S3)
mc mirror --overwrite --remove local/team-shared remote/team-shared-backup
# 3) Postgres 导出(自建时)；Supabase 云用其控制台或 pg_dump 连接串
# pg_dump "$PG_URL" | gzip > "$DST/pg.sql.gz"
```

## 11. 安全体系（代码层 → 传输层 → 服务器层 → 数据层）

### 11.1 认证与授权
- 现状静态 token 是「全员一把钥匙」，短期可用但**无法按人吊销/审计**。
  - 立即：token 用 `openssl rand -hex 32` 生成，`.env` 权限 600，定期轮换。
  - 中期：改为登录后签发**短期 JWT/一次性上传令牌**，网关校验用户+权限+过期时间。
- Supabase：**开 RLS**，前端 anon key 只能做被策略允许的操作；`service_role` 永不进前端。

### 11.2 上传安全（大文件场景重点）
- **扩展名白名单**（已做）+ **魔数校验**：读文件头字节核对真实类型，防「改后缀」绕过。
- **大小上限**：网关 + Nginx 双卡（已具备）。
- **禁危险类型**：`.html .htm .svg .js .exe .sh .php` 等，或统一强制 `attachment` 下载不内联。
- **病毒扫描**：上传落盘后 ClamAV 异步扫，命中即隔离 + 告警。
  ```bash
  sudo systemctl enable --now clamav-freshclam clamav-daemon
  # 网关 complete 后调用：clamdscan --no-summary <file>；非 0 则标记 quarantined
  ```
- **目录穿越**：已用 `safe_join_under` 防护，保持。
- **下载隔离**：`Content-Disposition: attachment` + `X-Content-Type-Options: nosniff`（Nginx 已配），
  理想再用**独立下载域名**（如 `dl.your-domain.com`）隔离 Cookie/同源风险。

### 11.3 传输与边界
- 全站 HTTPS，HSTS：`add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;`
- MinIO/网关/DB **只绑内网**，公网只有 Nginx 80/443。
- `fail2ban` 防 SSH/登录爆破；SSH 关密码登录、只用密钥、改非标端口。
- CORS 从 `*` 收敛到自己域名（改 `working_proxy._cors`）。

### 11.4 服务器层
- 专用 `appsvc` 用户跑服务，systemd 加固（见 5.3）。
- `.env`、`backup.key`、证书私钥权限 600，属主唯一。
- 自动安全更新：`unattended-upgrades`。
- 敏感路径 Nginx 拒绝（已配 `.git/.env/.py/.log`）。

### 11.5 审计与合规
- 保留审计日志与 Nginx 访问日志 **≥6 个月**（等保要求），异地归档。
- 关键操作（上传/删除/审批/登录）已进 `audit_event`，保持并定期审查。

## 14. index.html 过长：正确瘦身方案（核心技术债）

### 14.1 问题定性（不是「拆成多页面」）

| 现状（已核对，2026-07-14 仓库落地后） | 数字 |
|----------------|------|
| `123123/index.html` | ≈ **1350 行 / 0.4 MB**（原 3.1 万行 / 1.7 MB） |
| 外置核心脚本 | `js/app-core.js` + `app-legacy-a/b.js` + 既有业务 `js/*.js` |
| 全局样式 | `css/app-shell.css` |
| 懒加载模块 HTML | `modules/*.html` **34** 个（`data-lazy=1`） |
| 第三方库 | `vendor/` 本地化（xlsx/echarts/pdfjs/mammoth/tailwind/wangeditor/cropper/fonts） |

痛点（编辑器卡顿、冲突面、Review）已通过**机械外置**大幅缓解；运行时仍是 SPA，行为边界不变。  
备份原文件：`123123/index.html.bak-20260714`。

### 14.2 明确禁止的做法（会破坏现有功能）

对照仓库《拆分》评审结论，下列做法**禁止**作为瘦身手段：

1. **拆成多个独立 HTML（MPA）**：`currentUser` / `permissionMatrix` / 各业务数组 / 云同步轮询 / `showModule` 返回栈全部依赖单页内存常驻；跳页 = 丢状态 = 功能断裂。
2. **Nginx SSI 拼页面**：只能拼静态 HTML 碎片，**拆不掉**内联 JS，治不了病。
3. **一步上死 CSP（禁内联脚本）**：当前大量内联 `<script>` 与 `onclick=`，未外置完就收紧会白屏。
4. **为瘦身引入 Web Components + Shadow DOM 全站重写**：切断现有全局主题/CSS，成本远高于收益。
5. **再塞一套 CMS（如 Strapi）**：业务数据已在 Supabase KV，重复造轮子。

### 14.3 正确方向：管理端保持 SPA，「拆模块」不「拆页面」

```
目标结构（行为零变化，只改文件边界）

123123/
├── index.html              # 壳：登录框 + 侧栏 + 空 module 容器 + 核心脚本引用
│                           # 目标：从 ~3 万行降到 1千～2千行级
├── css/
│   └── app-shell.css       # 从 index 抽出的全局样式（可分批）
├── js/
│   ├── app-core.js         # ★新建：currentUser / 权限 / 云同步 / showModule / 事件总线
│   ├── application-center.js   # 已有
│   ├── home-dashboard-p3.js    # 已有
│   ├── notice-enhance.js       # 已有
│   └── …按模块继续抽
└── modules/                # ★新建：各模块 HTML 片段（按需懒加载）
    ├── weekly_report.html
    ├── notice_publish.html
    ├── member_archive.html
    └── …
```

运行时约定（与现状兼容）：

1. **仍是一个 `index.html` 入口**，登录后不整页跳转。
2. `showModule(id)` 逻辑保留在 `app-core.js`；首次进入某模块时：
   - 若 `#id` 容器为空 → `fetch('modules/'+id+'.html')` → 注入 → 再调该模块 `init*`；
   - 已注入则直接显示（与今天 `display` 切换一致）。
3. 共享状态**只**经 `app-core`（或 `window.AppCore`）读写，禁止模块互相偷改全局变量。
4. 每抽一个模块：本地冒烟该模块 CRUD / 权限 / 云同步 → 再抽下一个；**可随时回滚单文件**。

## 18. 风险与注意事项（务必先看）

1. **内存**：一次性大文件上传会整包进网关内存——大文件必须走分片或预签名直传，否则 OOM。
2. **磁盘**：大文件增长快，**没配用量告警 = 迟早写满导致全站宕**。数据盘独立 + LVM + 告警是硬要求。
3. **anon key**：Supabase 前端 key 一定暴露，安全靠 RLS，不靠隐藏。
4. **CSP**：别一步到位上死 CSP，会打断现有内联脚本/CDN；先 Report-Only，等 §14 外置完成后再转强制。
5. **备份要能恢复**：只备份不演练 = 没有备份。上线前跑一次异地恢复。
6. **绑定地址**：网关/MinIO 生产必须只绑 127.0.0.1 或内网，别裸奔 0.0.0.0 公网。
7. **index.html 瘦身**：严禁用 MPA/SSI「拆页面」；只许「拆模块」。每轮只动 1～2 个模块并回归，避免一次大爆炸重构。
